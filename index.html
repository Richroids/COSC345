Editor

For our editor we are after a heavy-weight, modern IDE that can support TypeScript. A heavy-weight IDE provides error checking and find usage functionality that can act as early debugging detection alongside time-saving features such as code completion. These requirements narrow down the options to some of the more well known IDE's that have had plug-ins written for TypeScript, and then some more specific editors that are designed to support TypeScript.

Editor Pros Cons
Visual Studio TypeScript is a Microsoft associated language and so it would appear logical that a Microsoft IDE would provide the most up to date support for TypseScript. Some Internet users have tested to see what IDE gets the most immediate updates when a new version of TypeScript is released. (source) while visual studio does get them rapidly, other IDE's appear a bit faster. Also visual studio runs only on Windows which could make it difficult for group members with mac based hardware.
Atom Atom is a text editor which prides itself on user customisation while maintaining features and usability. It has implemented support for TypseScript updates (using an extension download) faster than all the other editors also and has cross platform support. None of our group members have heard of atom before this project and some believe that our time spent up-skilling could be spent more valuably
Eclipse This is a modern, powerful IDE that all group members are somewhat familiar with. It has full TypseScript support with fast updates. It runs on windows and mac and additionally comes pre-installed on computers in Owheo. While one of two major IDE's that the University of Otago endorse, most group members are more familiar with NetBeans. Eclipse also doesn't come with any native GitHub repository support.
NetBeans Again, this is a modern IDE that all the group are familiar with but this time all group members have used this particular editor and know how to navigate it and use shortcuts. In addition, NetBeans can link directly to a GitHub repository making version control easier as well as being supported by the university computers. NetBeans can be slower to get updates for TypeScript and requires command line compilation.

//Choosing to use Eclipse, plug-in seems better than the one for NetBeans Overall, familiarity is the main requirement of our coders so long as the editor has full support for our chosen language and is a heavy-weight IDE. This has lead to our group chosing NetBeans as our editor of choice, despite the mentioned drawbacks. This remains a flexible decision and if any issues arise with TypeScript version support or anything not foreseen then we shall choose the next best editor based on our issues with NetBeans.

For the app we will be using Andriod Studio as it contains many features that are specifically engineered towards the development of an Android smart watch application.

Task Management

Much of the time spent before the second assignment will be spent up-skilling. As it is hard to know about things you do not yet know this will be hard to estimate. For this reason we plan to leave an arbitrary amount of time between each release which each group member can spend learning things as they discover it needs to be learnt. As there are experience differences in the group, work should be able to continue as the more inexperienced members get themselves up to speed. At the moment the planned time allowed for this is two weeks/10 days.

For task management it seems best to follow a scrum style organisation structure. This will give us clear goals with a measureable timeframe to help understand how our work process are advancing us to our end release goals. The intial phase of this is to create a list of user stories that define the release features from an end user perspective. This necessarily breaks down a big project into more managable pieces, that can evaluated and given more accurate time estimations compared to estimating as a whole.

The estimations made attempt to take into consideration that we are all university students with other papers and lives; unlike most software engineers this is not our full time job. To try and avoid the planning fallacy I have considered the size of weekly lab projects for other computer science papers and used this as a benchmark for the size of a 5 day project. Checking our tasks against these lab timeframes designed for students of our level and paper load should yield a more accurate assessment.

User Stories

  • As a user I want an analog clock to tell the time. -6 days
  • As a user I want upcoming tasks and events to overlay the analog clock to accessibly show events in the next 12 hours. -6 days
  • As a user I want to be able to sync my smart watch with a more heavy-weight app. -6 days
  • As a user I want the calendar to update as events pass to make sure only relevant information is displayed - 4 days
  • As a user I want to be able to view additional information about an event at a touch. - 6 days
  • As a user I want to be reminded of an event before it happens. - 4 days
  • As a user I want additional information about an event to be displayed at time of a reminder. -1 day
  • As a user I want a stylish display. - 8 days
  • As a user I want to be able to colour differentiate between different types of events. - 4 days

And as optional functionality if time permits:

  • As a user I want to be able to delete a calendar entry -6 days
  • As a user I want to be able to have different reminder vibrations for different event types to quickly distinguish the upcoming event -16 days
  • As a user I want to be able to view the weeks events -8 days
  • As a user I want to be able to view the months events -8 days

Excluding the extra functionality, this gives a total of 45 days, or 9 working weeks. This is bumped up to 55 days/11 working weeks when taking into account the time required to learn the tools needed for the project. The finished product assessment is in 65 working days excluding holidays and exam period (85 working days minus 20 days over the break) The extra 10 days at this stage will be used for report writing and any extra time the test team needs before release.

In the meantime, our draft code is expected 27 working days after this report. Our release goal for this is to have a designed (8 days), analog clock(6 days) with an overlaid calendar(6 days) linked to a heavy-weight app or repository(6 days) which should take a total of 26 working days to build, with testing running simultaneously. This does not leave any time allocated for report writing meaning that this will be done concurrently also.
